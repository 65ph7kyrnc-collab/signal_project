The Data Storage System is responsible for securely storing, organizing, and retrieving cardiovascular patient data generated by the simulator. Since the system processes real-time health information, the storage design needs to support efficient retrieval, maintainability, and data security.

The DataStorage class acts as the central storage component and manages Patient objects and their corresponding PatientRecord entries. Each Patient object owns multiple PatientRecord objects, which is why composition was used between these classes. A patient record should not exist without an associated patient.

I reused the existing project structure where patient data is stored inside a map of Patient objects. This design already supports efficient access to records grouped by patient ID. The PatientRecord class stores important information such as timestamps, measurement values, and record types. Including timestamps allows historical analysis and future trend detection.

To improve the architecture and satisfy the assignment requirements, I added the DataRetriever, AccessController, User, and DataRetentionPolicy classes. The DataRetriever class separates retrieval logic from storage logic, which improves cohesion and readability. The AccessController handles authorization checks and ensures that only permitted users can access patient records. This reflects privacy and security concerns that are important in healthcare systems.

The DataRetentionPolicy class models automatic deletion of outdated records, which supports scalability and long-term maintainability. Overall, the subsystem was designed to demonstrate modularity, clear class responsibilities, and secure handling of sensitive medical data.
